C\+Unit suite data type. \doxyref{C\+U\+\_\+\+Suite}{p.}{struct_c_u___suite} is a linked list of \doxyref{C\+U\+\_\+\+Test}{p.}{struct_c_u___test} containers. Each suite has a name, a count of registered unit tests, and a flag for whether the suite is active during test runs. It also holds pointers to optional initialization and cleanup functions. If non-\/\+N\+U\+LL, these are called before and after running the suite\textquotesingle{}s tests, respectively. In addition, the suite holds a pointer to the head of the linked list of associated \doxyref{C\+U\+\_\+\+Test}{p.}{struct_c_u___test} objects. Finally, pointers to the next and previous suites in the linked list are maintained.~\newline
~\newline


Generally, the linked list includes suites which are associated with each other in a \doxyref{C\+U\+\_\+\+Test\+Registry}{p.}{struct_c_u___test_registry}. As a result, suites are run in the order in which they are registered (see \doxyref{C\+U\+\_\+add\+\_\+suite()}{p.}{group___framework_ga94b8f1bbbd93b154a60bb1d43391b48a}).~\newline
~\newline


It is recommended that name of each \doxyref{C\+U\+\_\+\+Suite}{p.}{struct_c_u___suite} in a test registry have a unique name. Otherwise, only the first-\/registered suite having a given name will be accessible by name. There are no restrictions on the contained tests. This means that the same \doxyref{C\+U\+\_\+\+Test}{p.}{struct_c_u___test} could, in principle, be run more than once fron different suites.

{\footnotesize\ttfamily typedef struct \textbf{ C\+U\+\_\+\+Test\+Registry} \textbf{ C\+U\+\_\+\+Test\+Registry}}

C\+Unit test registry data type. C\+U\+\_\+\+Test\+Registy is the repository for suites containing unit tests. The test registry maintains a count of the number of \doxyref{C\+U\+\_\+\+Suite}{p.}{struct_c_u___suite} objects contained in the registry, as well as a count of the total number of \doxyref{C\+U\+\_\+\+Test}{p.}{struct_c_u___test} objects associated with those suites. It also holds a pointer to the head of the linked list of \doxyref{C\+U\+\_\+\+Suite}{p.}{struct_c_u___suite} objects. ~\newline
~\newline


With this structure, the user will normally add suites implictly to the internal test registry using \doxyref{C\+U\+\_\+add\+\_\+suite()}{p.}{group___framework_ga94b8f1bbbd93b154a60bb1d43391b48a}, and then add tests to each suite using \doxyref{C\+U\+\_\+add\+\_\+test()}{p.}{group___framework_gad9f198a8a5fa8cc6870c3c8be873869f}. Test runs are then initiated using one of the appropriate functions in Test\+Run.\+c via one of the user interfaces.~\newline
~\newline


Automatic creation and destruction of the internal registry and its objects is available using \doxyref{C\+U\+\_\+initialize\+\_\+registry()}{p.}{group___framework_ga3477b2d3e57c8f8aba36bbbdfa54920d} and \doxyref{C\+U\+\_\+cleanup\+\_\+registry()}{p.}{group___framework_ga183eb7d7f79e149c767237cfef52cb82}, respectively. For internal and testing purposes, the internal registry can be retrieved and assigned. Functions are also provided for creating and destroying independent registries.~\newline
~\newline


Note that earlier versions of C\+Unit also contained a pointer to a linked list of \doxyref{C\+U\+\_\+\+Failure\+Record}{p.}{struct_c_u___failure_record} objects (termed \+\_\+\+Test\+Results). This has been removed from theregistry and relocated to Test\+Run.\+c.

\doxysubsubsection{CU\_add\_suite()}
{\footnotesize\ttfamily C\+U\+\_\+\+E\+X\+P\+O\+RT \textbf{ C\+U\+\_\+p\+Suite} C\+U\+\_\+add\+\_\+suite (\begin{DoxyParamCaption}\item[{const char $\ast$}]{str\+Name,  }\item[{\textbf{ C\+U\+\_\+\+Initialize\+Func}}]{p\+Init,  }\item[{\textbf{ C\+U\+\_\+\+Cleanup\+Func}}]{p\+Clean }\end{DoxyParamCaption})}

Creates a new test suite and adds it to the test registry. This function creates a new test suite having the specified name and initialization/cleanup functions and adds it to the test registry. The new suite will be active and able to be executed during a test run. The test registry must be initialized before calling this function (checked by assertion). p\+Init and p\+Clean may be N\+U\+LL, in which case no corresponding initialization of cleanup function will be called when the suite is run. str\+Name may be empty (\char`\"{}\char`\"{}) but may not be N\+U\+LL.~\newline
~\newline


The return value is a pointer to the newly-\/created suite, or N\+U\+LL if there was a problem with the suite creation or addition. An error code is also set for the framework. Note that if the name specified for the new suite is a duplicate, the suite will be created and added but the error code will be set to C\+U\+E\+\_\+\+D\+U\+P\+\_\+\+S\+U\+I\+TE. The duplicate suite will not be accessible by name.~\newline
~\newline


N\+O\+TE -\/ the C\+U\+\_\+p\+Suite pointer returned should N\+OT BE F\+R\+E\+ED BY T\+HE U\+S\+ER. The suite is freed by the C\+Unit system when \doxyref{C\+U\+\_\+cleanup\+\_\+registry()}{p.}{group___framework_ga183eb7d7f79e149c767237cfef52cb82} is called. {\bfseries{This function must not be called during a test run (checked by assertion)}}. ~\newline
~\newline


\doxyref{C\+U\+\_\+add\+\_\+suite()}{p.}{group___framework_ga94b8f1bbbd93b154a60bb1d43391b48a} sets the following error codes\+:
\begin{DoxyItemize}
\item C\+U\+E\+\_\+\+S\+U\+C\+C\+E\+SS if no errors occurred.
\item C\+U\+E\+\_\+\+N\+O\+R\+E\+G\+I\+S\+T\+RY if the registry has not been initialized.
\item C\+U\+E\+\_\+\+N\+O\+\_\+\+S\+U\+I\+T\+E\+N\+A\+ME if str\+Name is N\+U\+LL.
\item C\+U\+E\+\_\+\+D\+U\+P\+\_\+\+S\+U\+I\+TE if a suite having str\+Name is already registered.
\item C\+U\+E\+\_\+\+N\+O\+M\+E\+M\+O\+RY if a memory allocation failed.
